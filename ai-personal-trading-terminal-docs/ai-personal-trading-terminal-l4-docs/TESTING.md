# 测试方案

## 1. 测试目标

确保系统在计算、风控、执行、异常处理方面可靠。

重点测试：

- 仓位计算；
- 风控拦截；
- 订单状态机；
- Dry Run；
- 交易所适配器；
- Kill Switch；
- AI输出约束。

---

## 2. 单元测试

### 必测模块

- position-sizing-engine；
- risk-engine；
- market-structure；
- decision-gate；
- execution-engine请求体生成；
- exchange-adapter精度处理。

### 仓位计算测试

覆盖：

- 不同权益；
- 不同风险比例；
- 不同止损距离；
- 不同杠杆；
- 不同数量精度。

---

## 3. 集成测试

### 3.1 计划检查流程

```text
创建计划 → 仓位计算 → 风控检查 → AI评估 → 决策门
```

### 3.2 订单预览流程

```text
计划通过 → 生成order intent → 生成请求体 → Dry Run → 保存日志
```

### 3.3 L4执行流程

```text
用户确认 → 提交订单 → 同步状态 → 设置止损止盈 → 日志
```

---

## 4. Dry Run测试

至少完成：

- 50次手动计划Dry Run；
- 50次自动计划Dry Run；
- 10次异常参数测试；
- 10次风控拦截测试；
- 10次重复提交测试。

---

## 5. 交易所测试

如果交易所提供模拟环境，优先使用模拟环境。

如果没有：

- 使用极小金额；
- 限制标的；
- 限价单；
- 逐仓；
- 必须止损；
- 全程人工监控。

---

## 6. AI测试

测试AI是否：

- 输出合法JSON；
- 不绕过风控；
- 不建议高杠杆；
- 不建议无止损；
- 能解释风控禁止原因。

---

## 7. 异常测试

必须模拟：

- WebSocket断开；
- 交易所API超时；
- 下单失败；
- 止损设置失败；
- 数据库写入失败；
- Kill Switch触发；
- 重复点击确认；
- 同一clientOid重复提交。

---

## 8. 验收测试

每个版本根据 `MVP_ACCEPTANCE.md` 执行验收。

未通过验收不得进入下一版本。
