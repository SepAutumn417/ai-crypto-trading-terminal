# 执行安全设计

## 1. 核心原则

执行系统必须以安全为第一目标。

> **Kill Switch 极性约定**：`kill_switch = true` 表示 Kill Switch 已激活（熔断态），禁止新开仓；`kill_switch = false` 表示恢复交易。下文"Kill Switch 激活"即指 `kill_switch = true`。

规则：

1. 没有风控通过，不允许执行；
2. 没有用户确认，不允许执行；
3. 没有止损，不允许执行；
4. Kill Switch 激活（`kill_switch = true`），不允许执行；
5. 交易所状态不确定，不允许执行；
6. 数据库无法写入，不允许执行；
7. API Key权限异常，不允许执行。

---

## 2. Kill Switch

### 2.1 作用

Kill Switch 是执行系统总开关。

> Kill Switch 的唯一事实源见 `KILL_SWITCH.md`。本节只描述执行侧的语义。

### 2.2 激活后允许（`kill_switch = true`）

- 查看行情；
- 查看结构；
- 创建计划；
- Dry Run；
- 查看持仓；
- 撤销订单；
- 手动确认平仓类减风险操作。

### 2.3 激活后禁止（`kill_switch = true`）

- 开新仓；
- 加仓；
- 自动提交订单；
- 进入订单确认页。

---

## 3. 自动触发Kill Switch条件

| 条件 | 动作 |
|---|---|
| WebSocket断开超过阈值 | 禁止新单 |
| 下单失败连续达到阈值 | 禁止执行 |
| 止损设置失败 | 立即Kill Switch |
| 数据库写入失败 | 禁止执行 |
| 当日亏损达到限制 | 禁止新单 |
| 风控配置异常 | 禁止执行 |
| 交易所API返回权限错误 | 禁止执行 |
| 本地时间与服务器时间偏差过大 | 禁止执行 |
| 检测到重复订单风险 | 冻结执行 |

---

## 4. API Key安全

- 使用环境变量或密钥管理服务；
- 数据库中不明文保存；
- 前端永不接触Secret；
- 交易Key不启用提现权限；
- 交易Key绑定IP；
- 日志中隐藏敏感字段；
- 支持手动轮换Key。

---

## 5. Dry Run安全流程

真实执行前必须完成Dry Run。

Dry Run检查：

- 订单参数完整；
- 风控通过；
- 精度正确；
- clientOid生成；
- 请求体可序列化；
- 订单预览正确；
- 执行日志可写入。

---

## 6. 风控安全

### 6.1 必须检查

- 风险金额；
- 杠杆；
- 止损；
- 盈亏比；
- 账户权益；
- 当日亏损；
- 连亏；
- 冷却期；
- 机会等级。

### 6.2 硬禁止

- 无止损；
- 超过最大风险；
- 超过最大杠杆；
- 当日亏损达限；
- Kill Switch关闭；
- 风控配置丢失。

---

## 7. 订单提交安全

订单提交前需要二次检查：

1. 计划未过期；
2. 当前价格未偏离过大；
3. 账户权益未变化过大；
4. 风控状态仍为允许；
5. 用户确认仍有效；
6. clientOid未使用；
7. 数据库事务已准备。

---

## 8. 止损安全

系统规则：

> 所有开仓订单必须有止损保护。

允许两种实现：

- 开仓时预设止损；
- 成交后立即设置止损计划单。

如果止损设置失败（TP_SL_FAILED）：

> 此规则在 `ORDER_LIFECYCLE.md §4.3`、`OPERATIONS.md §3.3`、本文三处统一为：**止损设置失败 → 立即触发 Kill Switch（`kill_switch = true`）+ 记录 CRITICAL 事件 + 禁止新单 + 提供补设止损 UI**。

- 立即触发 Kill Switch（`kill_switch = true`）；
- 记录 CRITICAL 级别 `system_events`；
- 通知用户；
- 禁止新订单；
- 提供补设止损确认 UI（一键补设止损）。

---

## 9. 异常恢复

系统重启后必须自动执行：

1. 读取本地未完成订单；
2. 查询交易所订单状态；
3. 查询持仓状态；
4. 对账；
5. 修复本地状态；
6. 对异常订单标记人工检查。

---

## 10. 日志要求

所有执行相关操作必须记录：

- 用户ID；
- 计划ID；
- 订单意图ID；
- 风控结果ID；
- AI评估ID；
- 请求体Hash；
- 交易所响应；
- 状态变化；
- 时间戳。
