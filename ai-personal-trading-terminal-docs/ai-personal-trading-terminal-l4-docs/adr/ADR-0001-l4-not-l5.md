# ADR-0001：选择 L4 而不是 L5

## 状态

Accepted（2026-07-07）

## 背景

本系统面向个人加密货币合约交易者，目标是减少手动填单错误与临场情绪干扰，同时保留用户对真实下单的最终控制权。

自动化等级（见 `AUTOMATION_LEVELS.md`）从 L1（纯手动）到 L5（全自动）递进。L5 全自动交易对系统的要求显著高于 L4：

- 需要经过长期实盘验证的策略正期望；
- 需要更强的异常处理与自恢复能力（无人值守）；
- 需要更严格的风控与资金管理（AI/策略可独立调仓）；
- 在部分司法辖区涉及合规与责任归属问题。

当前阶段（v1.0）系统的定位是**个人交易终端**，不是全自动交易机器人。用户希望减少重复操作，但不希望交出下单决策权。

## 决策

系统 v1.0 目标设为 **L4：用户确认后执行**。

系统不做 L5 全自动交易。

## 后果

**正面**：

- 用户保留最终确认权，风险可控；
- 风控模型更简单（无需处理策略自主加仓/反手）；
- UX 聚焦于"确认"单一动作，无需复杂的策略开关与置信度阈值；
- 合规风险低，用户对每笔订单负责；
- 复盘链路完整：用户确认是显式事件，可追溯。

**负面**：

- 执行速度慢于 L5（有人工确认延迟），不适合高频/套利场景；
- 需要设计 Order Preview 与二次确认 UX，前端复杂度增加；
- 用户不在场时无法捕捉机会（可接受，符合 L4 定位）。

**需新增的组件**：

- Execution Engine；
- Order Preview 页面；
- 用户确认流程（含二次确认，见 `AUTH.md §5`）；
- 订单状态机（见 `ORDER_LIFECYCLE.md`）；
- Kill Switch（见 `KILL_SWITCH.md`）。

## 备选方案

| 方案 | 拒绝原因 |
|---|---|
| **L3（Dry Run / 信号提示）** | 只生成信号不下单，无法减少手动填单错误，价值不足以支撑完整系统开发 |
| **L5（全自动）** | 需要策略长期实盘验证 + 更强异常自恢复 + 合规审查，超出 v1.0 范围；用户明确表示不愿交出下单权 |
| **L4.5（白名单标的自动执行）** | 引入"部分自动"的复杂性，风控边界模糊；v1.0 保持单一自动化等级，v1.0+ 再评估 |
