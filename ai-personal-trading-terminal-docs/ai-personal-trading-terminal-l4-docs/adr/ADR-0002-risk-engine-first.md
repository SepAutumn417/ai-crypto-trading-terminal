# ADR-0002：风控引擎优先

## 状态

Accepted（2026-07-07）

## 背景

系统涉及真实资金交易执行。在 L4 架构下，多个模块都可能与风控产生交集：

- **AI 评估**：LLM 可能产生幻觉，建议超限仓位、无止损交易或绕过既有规则；
- **前端**：若前端可直接调用执行 API，则网络层绕过风控；
- **执行引擎**：若执行引擎内联风控逻辑，规则变更需改代码而非配置，且难以单元测试；
- **候选计划生成**：结构识别算法可能产出不符合风控的计划。

历史上，交易系统的重大事故多源于"某个路径绕过了风控"（如 Knight Capital 2012、OKEx 2018 闪崩）。对于个人终端，一次风控失效即可导致爆仓。

## 决策

1. **风控引擎作为独立纯函数模块**（`packages/risk_engine`），输入为结构化数据，输出为 `RiskStatus`（ALLOW / WARN / REDUCE_RISK / BLOCK）；
2. **所有计划和订单必须经过风控**，无任何旁路；
3. **DecisionGate 综合风控与 AI**，但风控 BLOCK 永远 BLOCK，AI 不可覆盖（见 `AI_GUARDRAILS.md §5.1` 合并矩阵）；
4. 前端**不能**直接调用执行 API，必须经后端 DecisionGate → ExecutionEngine 链路；
5. 风控规则通过 `risk_config` 配置版本管理，规则变更走 `CONFIG_VERSIONING.md` 流程，不改代码。

## 后果

**正面**：

- 风控规则可单元测试（纯函数，无 IO），回归测试成本低；
- 风控结论可复盘：每笔订单关联 `risk_result_id`，历史可追溯；
- AI 无法绕过风控，安全边界清晰；
- 规则变更通过配置版本而非代码，支持 A/B 对比（见 `CONFIG_VERSIONING.md §8`）；
- 风控引擎可独立于执行引擎迭代，互不阻塞。

**负面**：

- 每次订单流经风控引入约 ≤ 50ms 延迟（见 `PRD.md §8.5`），对 L4 可接受；
- 规则变更需走配置版本流程 + 迁移，比硬编码慢；
- 风控引擎成为单点，其故障会阻塞所有交易（可接受：fail-closed 优于 fail-open）。

**强制约束**：

- 所有订单前必须生成 `risk_check` 记录并落库；
- DecisionGate 必须依赖 `risk_check`，不接受过期结果；
- 前端不得持有任何"已通过风控"的持久标记，每次确认都重新检查；
- 风控引擎故障时 ExecutionEngine fail-closed（禁止下单），不降级为"无风控执行"。

## 备选方案

| 方案 | 拒绝原因 |
|---|---|
| **执行引擎内联风控** | 规则不可独立测试，变更需改代码，难以复盘；风控与执行耦合，职责不清 |
| **AI 驱动风控** | LLM 非确定性，同一输入可能产出不同结论，无法满足"可复盘 + 可回归"；风控 BLOCK 可被 AI 覆盖，安全边界模糊 |
| **前端风控 + 后端复核** | 前端可被绕过（直接调 API），网络层无法保证风控强制执行 |
| **数据库触发器做风控** | 风控逻辑散落到 DB 层，不可测试，不可版本化，且触发器难以处理跨表业务规则 |
