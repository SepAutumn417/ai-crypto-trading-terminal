# ADR-0003：第一阶段先接 Bitget

## 状态

Accepted（2026-07-07）

## 背景

系统需要选择第一家交易所实现 L4 完整闭环（行情 → 计划 → 风控 → 确认 → 下单 → 止损 → 监控 → 复盘）。候选交易所包括 Bitget、Binance、OKX、Bybit 等。

多交易所并行适配的代价：

- 每家交易所的 REST/WebSocket API 签名、字段命名、订单状态码、精度规则、保证金模式都不同；
- 适配层需要统一的 `ExchangeAdapter` 抽象，抽象设计需要至少 2 家实现才能验证，初期只有 1 家时容易过度抽象；
- 测试矩阵随交易所数量线性增长；
- 个人开发者精力有限，并行适配会拖慢"打通真实下单"这一关键里程碑。

## 决策

第一阶段（v0.2 ~ v1.0）只接入 **Bitget USDT-M Futures**。

不在 v1.0 支持多交易所执行。

## 后果

**正面**：

- 聚焦单一适配器，最快打通真实下单闭环（v0.8 小额 L4）；
- `ExchangeAdapter` 抽象可推迟到第二家交易所接入时再固化，避免过度设计；
- 测试矩阵小，质量更易保证；
- Bitget 的合约交易、止盈止损、订单 WebSocket 能力满足 L4 需求。

**负面**：

- 单一交易所风险：若 Bitget 出现宕机/政策变动，系统不可用；
- 其他交易所用户需等待 v1.0+；
- Bitget 特有的 API 行为可能渗入业务层（需用适配器隔离）。

**强制约束**：

- 必须实现 `BitgetAdapter`，所有 Bitget 特有逻辑封装在适配器内；
- 业务层不得直接引用 Bitget 字段名、状态码或精度常量；
- 交易所参数（精度、最小下单量、合约面值）通过 `symbol_config` 配置管理，不硬编码；
- 适配器接口设计预留多交易所扩展点，但不提前实现。

## 备选方案

| 方案 | 拒绝原因 |
|---|---|
| **Binance 优先** | 部分地区用户访问受限，合规不确定性较高；API 限流较严 |
| **OKX 优先** | API 复杂度更高（多账户结构、保证金模式分支多），适配成本高于 Bitget |
| **Bybit 优先** | 合约产品线与 WebSocket 稳定性实测弱于 Bitget，社区文档较少 |
| **多交易所并行** | 个人开发者精力不足以同时适配多家；抽象层缺乏第二实现验证，易过度设计 |
| **交易所无关抽象优先** | 无具体实现支撑的抽象易偏离实际，应等第二家接入时重构 |
