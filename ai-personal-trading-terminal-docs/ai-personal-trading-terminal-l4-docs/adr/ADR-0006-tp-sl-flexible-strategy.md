# ADR-0006：止盈止损采用灵活双策略

## 状态

Accepted（2026-07-07）

## 背景

合约交易的止盈止损（TP/SL）有两种实现路径：

1. **开仓时预设 TP/SL**：在下开仓单的同时，通过交易所的 `reduceOnly` 计划单或带 TP/SL 参数的订单一并设置。优点是开仓即有保护，无保护窗口；缺点是部分交易所/订单类型不支持，且预设 TP/SL 与开仓单绑定时改单逻辑复杂。

2. **成交后立即设置 TP/SL 计划单**：开仓单成交后，单独提交 TP/SL 计划单。优点是适配所有交易所与订单类型；缺点是"成交 → TP/SL 设置成功"之间存在无保护窗口（通常 < 1s），若此窗口内 TP/SL 设置失败，仓位处于裸奔状态。

对于 L4 个人终端，止损是**硬性安全要求**（见 `EXECUTION_SAFETY.md §8`："所有开仓订单必须有止损保护"）。任何一种单一策略都有失败场景：

- 纯预设：Bitget 部分订单类型不支持带 TP/SL，强制限价单开仓时无法预设；
- 纯成交后设置：网络抖动导致 TP/SL 设置失败时，仓位裸奔，需触发 Kill Switch（见 `KILL_SWITCH.md §4`）。

## 决策

止盈止损采用**灵活双策略**：执行引擎根据交易所能力与订单类型自动选择，两种实现都允许。

- **优先**：开仓时预设 TP/SL（若交易所 + 订单类型支持）；
- **回退**：开仓单成交后立即设置 TP/SL 计划单（预设不支持时）。

无论哪种策略，**TP/SL 设置失败都触发 Kill Switch**（`kill_switch = true`），统一处理路径见 `ORDER_LIFECYCLE.md §4.3`。

## 后果

**正面**：

- 适配不同交易所能力差异，不强求统一行为；
- 预设成功时无保护窗口，安全性最高；
- 回退路径覆盖所有限价单场景；
- TP/SL 失败的应急处理统一（Kill Switch + 补设止损 UI），不分策略。

**负面**：

- 执行引擎需实现两条路径，测试面更大；
- "成交后设置"路径存在 < 1s 无保护窗口，依赖快速检测成交 + 快速设置；
- 复盘时需区分 TP/SL 是预设还是后设，日志需记录 `tp_sl_strategy` 字段。

**强制约束**：

- 订单状态机包含 `TP_SL_PLACING` → `TP_SL_PLACED` 中间态（见 `ORDER_LIFECYCLE.md §1`），无论哪种策略都经过此状态；
- `TP_SL_FAILED` 统一处理：立即 Kill Switch + CRITICAL 日志 + 禁新单 + 补设止损 UI；
- 执行引擎需在开仓前确定 `tp_sl_strategy`（`preset` / `post_fill`），写入 `order_intents`，不可中途切换；
- 成交后设置路径必须在成交事件后 **1 秒内**发起 TP/SL 设置请求，超时视为 FAILED。

## 备选方案

| 方案 | 拒绝原因 |
|---|---|
| **纯预设 TP/SL** | Bitget 部分订单类型（如限价单）不支持带 TP/SL 参数，强制市价单开仓限制交易策略 |
| **纯成交后设置** | 所有订单都经历 < 1s 无保护窗口，高频场景风险累积；预设本可避免的窗口被强加 |
| **仅市价单 + 预设** | 强制市价单开仓，用户无法限价建仓，滑点不可控，不符合 L4 用户体验 |
| **不设置 TP/SL，靠人工** | 违反 `EXECUTION_SAFETY.md §8` 硬性安全要求；人工依赖不可靠 |
